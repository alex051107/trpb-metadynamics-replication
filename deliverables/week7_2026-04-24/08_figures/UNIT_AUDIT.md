## 1 · The defect that triggered this audit

The previous canonical figure was generated by `plot_sz_si_sumhills.py` in
`transform="si_literal"` mode. That mode plotted the **raw** PLUMED `path.zzz`
column — which is in **Å² (length-squared, MSD)** — under the y-axis label
**"RMSD Deviation (Å)"**. The label says Angstrom; the values are

## 2 · X-axis — `path.s` (Open-to-Closed Path coordinate)

### 2.1 Definition (PLUMED)
`path.s` is the **Branduardi soft-min interpolated index** along the reference
path:

```
                  Sum_i  i · exp(-lambda · MSD_i(R))
    s(R)  =  ---------------------------------------
                    Sum_i  exp(-lambda · MSD_i(R))
```

where `i = 1, 2, ..., 15` indexes the 15 reference frames (MODEL 1 .. MODEL 15)
in `path_seqaligned_gromacs.pdb`, and `MSD_i(R)` is the per-Cα mean-squared
deviation of the current configuration `R` relative to MODEL `i`.

### 2.2 Units
**Dimensionless.** It is an interpolated integer index, not a physical length.

### 2.3 Range plotted
- Mathematically, `s ∈ [1, 15]` exactly when `R` matches MODEL 1 / MODEL 15.
- Practically `s` extends slightly outside `[1, 15]` because the soft-min is
  smooth, not a hard projection.
- Plot range: `0.8 ≤ s ≤ 15.2` — chosen to cover the full path with a small
  guard so the endpoint markers are visible.
- Tick locations: `[1, 3, 5, 7, 9, 11, 13, 15]` — odd integers from 1 to 15,
  matching the reference-frame indexing convention.

### 2.4 Reference values from production COLVAR
| Run | s_min | s_max | mean(s) |
|---|---:|---:|---:|
| Naive path baseline (job 45324928, 17.671 ns, 88,357 frames) | 1.0047 | 1.8958 | 1.1747 |
| Sequence-aligned pilot (job 45515869, 8.860 ns, 44,300 frames) | 1.0000 | 12.8665 | 4.9158 |
| Sequence-aligned `start.gro` projection at t = 0 | — | — | s = 7.0879 |

### 2.5 Why does `start.gro` project to s ≈ 7?
This is a **soft-min geometry artefact**, not a biological claim. Linear
interpolation between MODEL 1 (1WDW) and MODEL 15 (3CEP+5) produces middle
MODELs that sit at the geometric midpoint. For a configuration like
`start.gro` that is not strongly biased to either endpoint, the per-MODEL
RMSD spread is only 0.3–0.5 Å (1.30 Å to MODEL 7, 1.59 Å to MODEL 1, 1.76 Å
to MODEL 15). The soft-min then collapses to a near-uniform weighted average
of all 15 indices, giving `s ≈ N/2 = 7.5` ≈ 7. The complementary `z`
coordinate (off-path RMSD) confirms the configuration is **off the path**, so
no biological interpretation is warranted. Documented as the
`neighbor_msd_cv ≈ 0` limitation of linear-interpolation paths in the Belfast
PLUMED tutorial.

## 3 · Y-axis — `path.zzz` → RMSD Deviation (Å)

### 3.1 PLUMED first-principles unit chain
With `UNITS LENGTH=A ENERGY=kcal/mol` set in `plumed.dat`:

```
  z(R)  =  -(1 / lambda) · ln  Sum_i  exp(-lambda · MSD_i(R))
```

Dimensional analysis:
- `MSD_i(R)` has units of **length²** = **Å²**.
- The argument of `exp` must be dimensionless, so `lambda` has units of
  **Å⁻²** (inverse length-squared).
- Therefore `1/lambda` has units of **Å²**, and `z(R)` has units of **Å²**.

That is, `path.zzz` is a **squared distance** (an MSD), not a distance. To
plot it on an Angstrom axis, one must take the square root:

```
  RMSD_dev(R)  =  sqrt(path.zzz)            [units: Å]
```

This is what JACS 2019 Fig 3 / SI plot. Codex independent verdict
(CCB task `20260424-234500`) confirms by direct reading of the PLUMED
v2.9 PATH and PATHMSD docs:
- https://www.plumed.org/doc-v2.9/user-doc/html/_p_a_t_h.html
- https://www.plumed.org/doc-v2.9/user-doc/html/_p_a_t_h_m_s_d.html

### 3.2 The plotted axis
- **Label:** `RMSD Deviation (Å) = √p1.zzz` (canonical) — keeps the SI-Fig-3
  visual convention while making the transform explicit.
- **Range:** `0 ≤ RMSD_dev ≤ 1.7 Å` — matches JACS 2019 SI Fig 3 visual range.
- **Ticks:** standard matplotlib auto-ticks at 0.0, 0.4, 0.8, 1.2, 1.6.

### 3.3 Reference values verbatim
From COLVAR (raw col-3 in Å², sqrt in Å):

| Run | min(z²) [Å²] | max(z²) [Å²] | mean(z²) [Å²] | min(z) [Å] | max(z) [Å] | mean(z) [Å] |
|---|---:|---:|---:|---:|---:|---:|
| Naive | 0.1716 | 2.6336 | 1.5660 | 0.4142 | 1.6228 | 1.2175 |
| Sequence-aligned | 0.2446 | 2.5882 | 1.5211 | 0.4946 | 1.6088 | 1.2048 |

### 3.4 Two specific markers on the figure
| Marker | Raw (Å²) | sqrt (Å) | Plotted at (Å) | Source |
|---|---:|---:|---:|---|
| `start.gro` star (panel b) | 1.6814 | 1.2967 | **1.30** | seqaligned COLVAR row at t=0 |
| `UPPER_WALLS AT=2.5` guide (both panels, white dashed) | 2.5000 | 1.5811 | **1.58** | `plumed.dat` upper-wall config, in raw p1.zzz units |

### 3.5 The `UPPER_WALLS` is configured in raw units
The PLUMED restraint reads:
```
  UPPER_WALLS ARG=p1.zzz AT=2.5 KAPPA=800 LABEL=uwall
```
which means the wall acts at `p1.zzz = 2.5 Å²` in the **raw** PLUMED variable.
On the unit-correct (Angstrom) y-axis we draw the same wall at
`sqrt(2.5) = 1.5811 Å`. Both representations describe the same physical
restraint; the conversion is explicit.

## 4 · Colorbar — Free energy (ΔG, kcal/mol)

### 4.1 Source
`fes_initial_*_sumhills.dat` — output of PLUMED post-processing tool
`plumed sum_hills`, run on Longleaf:

```
  plumed sum_hills --hills HILLS \
      --min 0.5,0.0 --max 15.5,2.8 \
      --bin 300,140 --mintozero --outfile fes_*.dat
```

`sum_hills` integrates the deposited Gaussian bias in MetaD and returns
`F(s, z) = -V_bias(s, z) / (1 - 1/γ)` where `γ` is the bias factor; with
`--mintozero`, the minimum is set to 0 so all values are non-negative.

### 4.2 Units
**kcal/mol**, propagated from the `UNITS ENERGY=kcal/mol` setting in
`plumed.dat`. PLUMED applies this consistently through MetaD bias
deposition, sum_hills integration, and `.dat` file output.

### 4.3 Range plotted
- Plot clip: `0 ≤ ΔG ≤ 14 kcal/mol` — matches JACS 2019 main-paper Fig 3
  colorbar range. NaN / Inf values from sparsely sampled regions are
  replaced with `FES_MAX = 14.0` so they render at the colorbar maximum.
- Levels: 29 evenly spaced contour fills from 0 to 14.
- Black contours: every 2 kcal/mol from 2 to 14, line width 0.25, alpha 0.35.
- Colormap: `turbo` (matplotlib perceptually-uniform default for diverging-ish
  scalar data). Codex callout #5 confirmed this is fine for SI-style
  diagnostic plots.

### 4.4 Hedge — these are NOT converged WT FES surfaces
Subtitle on the canonical figure reads:
> Single-walker fallback variant (HEIGHT=0.3, BIASFACTOR=15, KAPPA=800);
> diagnostic surfaces clipped to SI 0-14 kcal/mol — not a converged WT FES.

This explicitly tells reviewers that the depth scale is borrowed for visual
comparability, not validated by convergence. SI's converged FES required
500 ns / 10 walkers; we have 8.86 ns / 1 walker.

## 5 · Why peaks/valleys differ from SI

| Feature | SI (JACS 2019 Fig 3, WT) | Our pilot (panel b) | Why the difference |
|---|---|---|---|
| O-state basin (s ≈ 1) | ~0–2 kcal/mol depth | shallow, walker leaves at t ≈ 4920 ps | Insufficient sampling; biased toward walker history |
| PC-state basin (s ≈ 7) | ~3–5 kcal/mol depth | not resolved as a basin | 8.86 ns walker spent 9.92% time at s<1.25 and 5.4% at s≥10; PC region under-sampled |
| C-state basin (s ≈ 12) | ~0–3 kcal/mol depth | partially explored, max s = 12.87 (transient) | A single ≈120-ps transient near the wall, not a sustained basin |
| Barrier heights (O↔PC, PC↔C) | ~6–8 kcal/mol | not resolvable | MetaD has not yet flooded these wells; need 10-walker production |
| Off-path z-axis spread | ~0–1.5 Å | ~0.4–1.6 Å | Comparable; UPPER_WALLS at 1.58 Å enforces physical similarity |

**Bottom line:** the geometric features (basin centers in `s`, off-path
spread in `z`) match SI within expected ranges; the energetic features
(basin depths, barriers) require 10-walker production to match.

## 6 · Parameters that DO NOT match Miguel's primary 10-walker contract

This was Codex's callout #5 and is the most important hedge for the slide
audience. Empirically verified against the active `path_seqaligned/plumed.dat`
AND the HILLS file column 8 (biasf), column 7 (deposited height):

|---|---:|---:|---|
| HEIGHT (kcal/mol) | 0.15 | **0.3** | Pilot uses Miguel's *aggressive single-walker fallback*. Verified from HILLS col-7: first deposit height ≈ 0.32 ≈ HEIGHT × BF/(BF-1) = 0.3 × 15/14 ✓ |
| BIASFACTOR | 10 | **15** | Pilot fallback. Verified from HILLS col-8 = 15.0 across all 4429 deposits |
| PACE (steps) | 1000 | 1000 | ✅ matches (re-checked Miguel template) |
| KAPPA (kcal/mol/Å⁴) | 800 | 800 | ✅ matches Miguel primary exactly |
| ADAPTIVE | DIFF | DIFF | ✅ matches |
| TEMP (K) | 350.0 | 350.0 | ✅ matches |
| LAMBDA (Å⁻²) | 80 | 80 (seqaligned) / 3.77 (naive) | seqaligned panel matches Miguel exactly; naive panel uses path-derived self-consistent value (Branduardi heuristic on 26× sparser path) |
| UPPER_WALLS AT | 2.5 (Å²) | 2.5 (Å²) | ✅ matches; on the canonical Å y-axis this displays at √2.5 = 1.58 Å |
| GRID_BIN | 300,100 | 300,100 | ✅ matches (on-the-fly bias grid; sum_hills uses a separate 300,140 reconstruction grid) |
| WALKERS_N | 10 | 1 | **Different by design** — pilot is a single-walker validation, not production |
| WALKERS_DIR | `../HILLS_DIR` | (none) | Single walker, no shared HILLS directory |

**Bottom line on Miguel-compliance:** the pilot is a **single-walker
fallback variant** of the Miguel contract — same path, same LAMBDA, same
KAPPA, same wall location, same ADAPTIVE/TEMP/GRID/PACE — only HEIGHT
(0.3 vs 0.15), BIASFACTOR (15 vs 10), and walker count (1 vs 10) differ.
Miguel labelled the `0.3 / 15` variant explicitly as an "aggressive
fallback" to be used when the primary set fails to converge or when only
short single-walker validation is feasible. Calling this run "identical
Miguel parameters" is overclaim; calling it "single-walker fallback variant
of the Miguel contract" is accurate. The figure subtitle, slide deck, and
TechDoc all use the latter phrasing.

## 7 · X-vs-Y consistency check (random sanity probe)

If a reviewer asks: "do the values plotted on the y-axis (sqrt-Å) actually
correspond to the same row indexed on the x-axis?" — yes. Both columns are
read from the same row of the same `fes_initial_*_sumhills.dat` file,
reshaped via `read_sumhills` such that `s_grid`, `z_grid`, `f_grid` all share
the same `(NZ, NS)` shape. The sqrt is applied element-wise to `z_grid` only;
`s_grid` is untouched. See `plot_sz_si_sumhills.py:52-65`.
