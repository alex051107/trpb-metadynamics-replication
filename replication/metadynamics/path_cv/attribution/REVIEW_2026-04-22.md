## Findings

### BLOCKING — `occupied_bins` threshold is density-confounded

- Files: `compare_projections.py:101` (bin counter), `compare_projections.py:141` (`AREA_CHEMISTRY_RATIO` check), `run_attribution.sh:44,50` (window/stride defaults).
- Threshold as written: Aex1/Ain occupied-bin ratio ≥ 2.0 ⇒ "chemistry-prior-strengthened."
- Under current defaults, Aex1 will structurally have fewer frames than Ain over the same time window, capping its bin count. The ratio could read as < 1.0 even if Aex1 is genuinely exploring a larger (s, z) area, purely because it's sampled 10× less often.
- **Do not publish the verdict tag from this rule until fixed.**
- Fix options:
  1. Stride the denser line to match the sparser: e.g. Ain stride 10 so both land at 100 ps/frame. Simple, loses information.
  2. Compute bin coverage density-normalized: `occupied_bins / sqrt(n_frames)` or plateau-of-coverage-curve metric. Robust, one-afternoon change.
  3. Replace raw-count threshold with a saturation-based metric (what fraction of 95 %-enclosed (s, z) area does each line fill).

### WARN — `plumed driver --timestep 0.002` cosmetic, but COLVAR time column is not physical

- Files: `project_to_path.py:77,233`.
- Driver receives a fake 0.002 ps timestep while the PDBs are frame-per-model exports of 10–100 ps snapshots. Comparator ignores time column, so no numerical impact, but the emitted `COLVAR_proj` time is meaningless. If anyone reuses the COLVAR for time-dependent analysis, they will be misled.
- Fix: set `--timestep` from the per-line `dt` (read from `u.trajectory.dt`) and emit it into the `# provenance:` line.

### WARN — geometry rule is s-only

- File: `compare_projections.py:149`.
- Current geometry-strengthened rule uses |Δ median_s|, IQR_s ratio, |Δ p95(s)|. Does not check z.
- Our probe-sweep failure mode (FP-024/025/027) is σ-floor pressure on both s and z. If Aex1 broadens in z without changing s, the rule will still call it geometry-driven.
- Fix: add a z-statistics twin rule and require both to pass before tagging "geometry-prior-strengthened."

### WARN — PDB atom-name field non-canonical

- File: `project_to_path.py:195`.
- Codex verified the emitted line is 80 chars with field columns correct enough for PLUMED to parse, but `line[12:16]=='CA  '` is looser than the standard PDB alignment for one-letter elements (should be ' CA ' with leading space).
- Fix: emit ` CA ` (leading space). Low priority — PLUMED parses either form on current path_gromacs.pdb (which uses the same loose form).

### NIT — selection error messages

- File: `project_to_path.py:136`.
- On selection failure, error doesn't distinguish "0 matches" from ">1 matches". Still fails loudly.

## Cleared (no blocker)

- Atom selection: `resid 97:184,282:305 and name CA` resolves uniquely for all 112 residues on **both** live topologies (Ain parm7, Aex1 pdb). No silent 5DW0 numbering offset.
- MDAnalysis can read the `.nc` (AMBER NetCDF) and `.dcd` (OpenMM DCD) as-is; `ts.time` returns ps reliably on these specific files.
- Driver-local per-MODEL serial renumbering matches the FP-030 pattern.
- 80-char ATOM line formatting lands in the PLUMED-parseable fixed columns.

## Implication for the 2×2 pilot matrix

The **Ain-alone z-drift check** (project existing Ain 500 ns cMD onto current path, plot z vs t) is single-line and does not invoke the verdict tag, so the BLOCKING issue does not affect it. Run Ain-alone first; fix `occupied_bins` before running the Ain-vs-Aex1 comparison.
